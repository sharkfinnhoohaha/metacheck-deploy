# MetaCheck Label-Tier Build — Scope & Phased Plan

Adversarial design workflow (map → design → harden → plan). Internal tier is `team` (= "Label" in UI). All gating reuses the existing single-string `users.tier` model; nothing adopts Clerk Organizations.

> **PR-exception work**: touches schema + billing-adjacent surfaces → builds on a feature branch with a PR, not direct-to-main.

## Phased plan

**Total effort:** XL overall (roughly 5-7 focused engineering days for one developer, dominated by Phase 3). Per-phase: Phase 0 = M (consolidated migration + getUserTier seam/memo, ~0.5-1 day); Phase 1 Support = M (~1 day, self-contained); Phase 2 Public API = L (~1.5-2 days, the mandatory fixFlow extraction is the bulk and pays back in both routes); Phase 3 Multi-seat = XL (~2-3 days — hottest auth path + 2 history pages + releases FK/reads/writes + 2 RPCs + invite/email infra that does not exist in the repo; this single phase is the reason the program is XL not L); Phase 4 Custom rules = M (~1 day, isolated to the validation surface). Phases 1, 2, and 4 can ship to production independently and incrementally; Phase 3 should ship last and be feature-flagged/beta-gated given its blast radius on the shared auth path.

**Sequencing:** Ordering is driven by dependency + value, and one hard structural fact: all four designs each proposed a file literally named 005_*.sql — they collide and several touch shared surfaces (the getUserTier seam, the releases FK). So Phase 0 consolidates them into ONE 005_label.sql and fixes the getUserTier seam first. Every phase is independently shippable behind the existing tier==='team' (Label) / tier!=='free' (support) gate, and each is safe to deploy alone because the migration is additive/idempotent and unused tables cost nothing.\n\nPhase 1 (Support) is first by VALUE-per-effort: it is genuinely standalone (touches no other feature), needs only support_tickets, and is the quickest credible Label/Pro win — ship it while the rest is in flight. Phase 2 (Public API) is next because it is self-contained, only READS tier, and — critically — forces the MANDATORY extraction of lib/ai/fixFlow.ts, which both Phase 2's /api/v1/ai-fix and the existing in-app route then share; doing it here prevents two copies of the interlocking credit/refund logic from drifting (it has already taken two audit-pass bug fixes). Phase 3 (Multi-seat) is deliberately AFTER the API because it is the XL lift that mutates the hottest auth path (getUserTier effective-tier CLAMP), rewrites two history pages + the releases route, changes the releases FK, adds 2 RPCs, and needs net-new email/invite infra; it benefits from the API phase having already proven the request-memo around getUserTier under load. Phase 4 (Custom rules) is last among the features: it is M-effort and isolated to the validation surface, but it depends on the engine being importable/stable (the API phase's purity guard) and is pure upside polish on the Label value prop.\n\nHard rules threaded through all phases: tier is written ONLY by Stripe/PayPal webhooks (every feature READS tier, never writes it); keep the {data,error} shape and Zod-bounded arrays; new PUBLIC routes (/api/v1, /team/accept) MUST be added to middleware isPublicRoute, new AUTHED routes (/api/keys, /api/rule-config, /api/support) must NOT; cost-bearing public endpoints FAIL CLOSED on rate limiting; all DB access via supabaseAdmin scoped by clerk_id/team_id with RLS as defense-in-depth only.\n\nTwo cross-cutting decisions to lock before their phase ships, not implicitly: (a) Phase 3's AI quota — uncapped 5x1500=7500 AI calls/mo on a $29 sub is a real cost gap; pick a lowered per-member team limit or a shared-pool consume_ai_call_team RPC. (b) Phase 2's rule-id contract — fine as an UNSTABLE beta contract now, but freeze a RULE_IDS catalogue + CI check before any external customer hits /api/v1/validate (GA blocker). Operator action item surfaced in chat (not buried): 005_label.sql must be applied in the Supabase SQL editor after 001-004 before/with the first Label deploy.

## Feature designs (hardened)

### Label-tier multi-seat / team accounts (Label subscription shared by up to 5 members; per-member identity + shared release history)  _(effort: XL)_

**Recommended approach:** KEEP the core thesis — custom `teams` + `team_members` tables keyed on `clerk_id`, do NOT adopt Clerk Organizations. That part is correctly reasoned and fits the codebase: identity is the Clerk `userId` string stored as `clerk_id` (the FK target for usage/releases), tier is a per-user column written only by billing webhooks, and nothing in the app reads org context. Adopting Orgs would re-architect middleware, app/(app)/layout.tsx, every API route, and all of lib/auth — unjustified for a fixed 5-seat promise.

But the submitted design has FOUR load-bearing defects that must be fixed:

1) LAPSE/DOWNGRADE SEMANTICS ARE WRONG. The design inherits "the OWNER's live users.tier" verbatim. If the owner downgrades team→pro (not cancel), members would inherit 'pro' and gain 300 AI calls — a privilege LEAK, not a revocation. Effective tier for a member must CLAMP: a member inherits 'team' IFF owner.tier === 'team', else 'free'. Never inherit 'pro' or any other value through membership.

2) THE HISTORY GATE DOES NOT GO THROUGH getUserTier. The design claims changing getUserTier 'makes history work unchanged'. FALSE — app/(app)/history/page.tsx:18-24 reads users.tier INLINE via supabaseAdmin (.select('tier').eq('clerk_id', userId)) and gates `=== 'free'`, bypassing getUserTier entirely. So the effective-tier change alone leaves members locked out of history. history/page.tsx (the tier read at :18-24 AND the release query at :47-52) and history/[id]/page.tsx (:28-35 .eq('clerk_id',userId)) BOTH need explicit edits. This is the single most likely silent break.

3) THE SEAT/UNIQUENESS CONSTRAINTS AS WRITTEN ARE INVALID/INSUFFICIENT. `UNIQUE(team_id, clerk_id) WHERE ...` and `UNIQUE(clerk_id) WHERE ... AND status='active'` cannot be expressed as table CONSTRAINTs with a WHERE — they must be partial CREATE UNIQUE INDEX statements (the 004 users_paypal_sub_unique pattern). More importantly, seat enforcement must be ATOMIC inside the RPC at BOTH invite-time AND accept-time (a pending invite issued while a seat was free can be accepted after the seat filled), mirroring how consume_ai_call enforces the limit inside the UPDATE, not in app code.

4) EFFORT IS UNDERSTATED. 'L' is wrong. This modifies the hottest auth path (getUserTier, called by every gated action), adds 4-5 routes, rewrites 2 history pages + the releases route reads/writes, adds owner/member UI, a migration with 2 RPCs, AND requires email infrastructure that does not exist in the repo. Realistic effort: XL.

Recommended shape (corrected): A Label subscriber (tier='team' — keep this internal value, never introduce 'label' or getUserTier coerces it to 'free') becomes an OWNER. A `teams` row is created lazily on first team-settings visit OR (preferably) inside the Stripe/PayPal webhook when tier is set to 'team', so the team exists before any invite. Owner invites up to N-1 others by email; an invite is a pending team_members row (clerk_id NULL) with a hashed token. On accept (post Clerk sign-in), ensureUser runs first, then an atomic accept_team_invite RPC binds clerk_id, sets status='active', and re-checks the seat cap in one statement. Membership grants EFFECTIVE tier='team' ONLY while owner.tier==='team'. Shared history is delivered by stamping releases.team_id at insert and scoping history reads by team_id (keeping releases.clerk_id for author attribution). When the owner cancels, the Stripe webhook sets owner tier='free' (existing code, zero new webhook work) and every member's next getUserTier returns 'free'.

**Alternatives considered:**
- _Clerk Organizations (orgId membership + Clerk-hosted invites/roles/emails)_ — Gets invite UI, role management, member lists, AND invite-email delivery for free — which is attractive given the repo has no email infra. But it's a heavy re-architecture: identity everywhere is the single Clerk userId stored as clerk_id (FK for usage/releases), tier lives per-user written by billing webhooks, and nothing uses org context. Adopting Orgs means threading orgId through middleware, app/(app)/layout.tsx, every API route, the entire lib/auth tier model, and reconciling Clerk org-billing with Stripe-as-source-of-truth. High risk/effort/vendor-coupling for a fixed 5-seat promise. Worth REVISITING only if multiple team features land (shared profiles/custom rules/SSO) — then Org primitives amortize.
- _Shared single clerk_id (everyone logs into one Label account)_ — Zero schema change. Breaks the per-member-identity requirement (no attribution, no individual profiles), violates Clerk's session model, and makes AI/usage accounting meaningless. Rejected as dishonest delivery.
- _Replace releases.clerk_id with team_id as the only owner key_ — Cleaner shared catalog, but destroys per-member attribution, breaks every .eq('clerk_id', userId) read and the FK/cascade semantics, and needs a risky backfill. Keeping clerk_id (author) + adding team_id (scope) is strictly safer and additive.
- _Per-seat metered Stripe billing_ — Better unit economics and directly addresses the AI-cost exposure, but needs new price IDs, Stripe quantity sync in checkout + webhook, and proration — far more billing surface than '5 included at $29'. Defer until the flat model proves insufficient.
- _Shared team AI pool (team-scoped consume_ai_call_team RPC keyed by team_id)_ — Directly fixes the 5×1500=7500 cost gap by metering one shared monthly AI budget per team instead of per member. Slightly more DB work (a team_usage row or team_id-keyed usage) and changes reserveAiCall to resolve a team_id, but it is the honest cost control for a single shared subscription. Strongly consider for v1 over per-member quotas.

**Schema changes:**
- NEW migration supabase/migrations/005_teams.sql — additive + idempotent (CREATE TABLE IF NOT EXISTS / ADD COLUMN IF NOT EXISTS / CREATE OR REPLACE FUNCTION / CREATE UNIQUE INDEX IF NOT EXISTS), per the 003/004 contract.
- TABLE teams: id UUID PK DEFAULT gen_random_uuid(); owner_clerk_id TEXT NOT NULL REFERENCES users(clerk_id) ON DELETE CASCADE; seat_limit INTEGER NOT NULL DEFAULT 5; created_at TIMESTAMPTZ DEFAULT now(). One team per owner via a PARTIAL UNIQUE INDEX (not an inline UNIQUE-with-WHERE constraint, which is invalid): CREATE UNIQUE INDEX IF NOT EXISTS teams_owner_unique ON teams(owner_clerk_id). (owner_clerk_id is already non-null so a plain unique index suffices, mirroring users_paypal_sub_unique style.)
- TABLE team_members: id UUID PK DEFAULT gen_random_uuid(); team_id UUID NOT NULL REFERENCES teams(id) ON DELETE CASCADE; clerk_id TEXT REFERENCES users(clerk_id) ON DELETE SET NULL (NULLABLE until accept; use SET NULL — see releases note — so removing a Clerk user does not silently shrink the active-seat audit row, and never CASCADE-deletes a team row); invited_email TEXT NOT NULL; role TEXT NOT NULL DEFAULT 'member' (free-text 'owner'|'member', no enum, consistent with users.tier); status TEXT NOT NULL DEFAULT 'pending' ('pending'|'active'|'revoked'); invite_token_hash TEXT (store ONLY a SHA-256 hash, never the raw token; NULL after accept); created_at TIMESTAMPTZ DEFAULT now(); accepted_at TIMESTAMPTZ.
- team_members partial unique indexes (as CREATE UNIQUE INDEX ... WHERE, the only valid form): (a) ux_team_members_active_clerk ON team_members(clerk_id) WHERE clerk_id IS NOT NULL AND status='active' — a clerk_id is in at most one active team; (b) ux_team_members_team_clerk ON team_members(team_id, clerk_id) WHERE clerk_id IS NOT NULL — no double-join to the same team. Plus btree indexes idx_team_members_team ON (team_id) and idx_team_members_clerk ON (clerk_id) WHERE clerk_id IS NOT NULL.
- ALTER TABLE releases ADD COLUMN IF NOT EXISTS team_id UUID REFERENCES teams(id) ON DELETE SET NULL. Keep releases.clerk_id (author). CRITICAL FK FIX: releases.clerk_id is ON DELETE CASCADE today (001_schema.sql:24) — so a member deleting their Clerk account CASCADE-deletes their releases out of the shared catalog. For label catalog integrity, change releases.clerk_id FK to ON DELETE SET NULL (drop+recreate the constraint in 005; make clerk_id nullable) so a departed member's releases survive, scoped by team_id, with author shown as 'former member'. (If author attribution must be preserved post-deletion, store an author_name TEXT snapshot at insert instead.) Index idx_releases_team ON releases(team_id).
- RPC claim_team_seat(p_team_id UUID, p_email TEXT, p_token_hash TEXT) RETURNS UUID — in ONE statement: count non-revoked rows for the team, and INSERT a pending row only if count < (SELECT seat_limit FROM teams WHERE id=p_team_id); return the new id, or NULL when full (NULL-on-full contract, mirroring consume_ai_call / consume_credit). Do the count+insert atomically (INSERT ... SELECT ... WHERE (SELECT count ...) < seat_limit) so two concurrent invites cannot both take the last seat.
- RPC accept_team_invite(p_token_hash TEXT, p_clerk_id TEXT) RETURNS UUID — atomically: find the pending row by token hash, re-check seat cap (a seat could have filled since the invite was issued), bind clerk_id, set status='active', accepted_at=now(), invite_token_hash=NULL; return team_id on success or NULL (expired/full/already-used). This closes the accept-time race the app cannot close safely on its own.
- ENABLE ROW LEVEL SECURITY on teams + team_members with NO policy (service-role only, exactly like webhook_events / the existing 3 tables) — RLS is defense-in-depth; the app uses supabaseAdmin because Clerk JWTs never reach Postgres.

**Gating:** Priority is derived server-side from the existing tier model, never the client. The route calls getUserTier(userId) (lib/auth/index.ts:42 — the same helper canValidate/canUseAI use) and sets priority='priority'/priority_rank=0 for tier 'pro' OR 'team' (matching the 'Pro + Label' scope) and 'standard'/1 for 'free'. The /support page's SLA copy comes from the same getUserTier read. NO change to MONTHLY_*_LIMITS and NO change to the Tier union — tier stays exactly 'free'|'pro'|'team' ('team'=Label); we only READ tier. Because tier is written EXCLUSIVELY by Stripe/PayPal webhooks (the source of truth, per CLAUDE.md), an upgrade automatically grants priority on the next ticket and a downgrade (webhook -> tier='free') reverts it, with zero extra wiring. Crucially, NO race-safe RPC (consume_ai_call style) is needed here: that pattern exists to ATOMICALLY DECREMENT a finite quota under concurrency (the free AI taste / credits), where a read-then-write race lets a user over-spend. Support priority is a pure READ of tier with no decrement and no shared counter to race on — a plain getUserTier read is correct and sufficient. The only concurrency concern is duplicate-ticket spam, which the 5/hour DB-count guard (plus Upstash fast path) handles; that guard does NOT need to be atomic because a 1-2 ticket overshoot under a race is harmless (worst case the operator sees one extra ticket), unlike over-spending paid quota. Route protection: Clerk middleware deny-by-default + (app)/layout.tsx redirect to /sign-in — no allowlist edit. DB access: supabaseAdmin scoped by .eq('clerk_id', userId) for the user's own writes/spam-count, unscoped+isAdminUser-gated for the inbox — identical to the releases-vs-admin pattern.

**Billing impact:** None to the billing system. Only operational cost is human support time (out of code scope) and, optionally, an Upstash limiter (already env-gated) and a Slack/Discord incoming webhook (free).
